# HoneyMesh Operations Runbook

## 1. Pre-deployment approval

Record:

```text
system owner
security owner
privacy owner
legal review status
deployment mode
authorized network ranges
data retention period
alert recipients
kill-switch operator
change ticket
fixture version
policy version
```

Internet-facing deployment should receive privacy and legal review appropriate to the jurisdiction and organization.

---

## 2. Build and isolation checks

Before every release:

1. Build from a clean, pinned source revision.
2. Verify the image contains no production credentials.
3. Verify no production hostnames appear in fixtures or configuration.
4. Confirm default-deny egress.
5. Confirm metadata-service access is blocked.
6. Confirm the filesystem is read-only except for an ephemeral volume.
7. Confirm no host or container-runtime socket is mounted.
8. Confirm request methods and body sizes are limited.
9. Confirm all decoy actions are inert.
10. Confirm the kill switch works without model services.

---

## 3. Canary lifecycle

### Creation

- Mint per-session canaries using the `hm_demo_` namespace.
- Store only a keyed hash in routine logs.
- Map the identifier to fixture and session in a restricted canary registry.
- Set an expiration no longer than the evidence-retention window.

### Rotation

Rotate:

- at deployment;
- after a P1 event;
- after fixture changes;
- after operator access to the registry;
- at the end of each lab campaign.

### Revocation

Canaries are nonfunctional by design. Revocation means deleting registry mappings and configuring the sink to reject or ignore expired identifiers.

## 4. Normal monitoring

Dashboard panels:

```text
sessions by disposition
requests per second
distinct routes per session
semantic duplicate ratio
sensitive-route attempts
canary events
terminal-response retries
classifier latency and tokens
queue depth
CPU and memory
benign spillover
false-positive review outcomes
```

Run a synthetic benign probe at least every five minutes in production-like deployments. Its queue must be separate from adversarial sessions.

---

## 5. Alert response

### P1

1. Verify the alert originated from the honeypot.
2. Activate the session kill switch.
3. Confirm network isolation remains intact.
4. Preserve signed logs and deployment metadata.
5. Rotate canaries.
6. Check for any production dependency or shared identity.
7. Notify the incident lead.
8. Rebuild the honeypot from a clean image before reopening.
9. Correlate with other defensive telemetry according to approved policy.

### P2

1. Quarantine the session.
2. Preserve the event sequence and fixture versions.
3. Review for false positives.
4. Check whether the same synthetic identifier appears in other sessions.
5. Adjust scoring only after replaying the event against benign baselines.

### P3

1. Keep the session throttled or quarantined.
2. Review aggregated patterns rather than one request.
3. Add to periodic threat-hunting analysis.
4. Avoid broad blocks without corroboration.

---

## 6. Evidence package

Preserve:

```text
deployment and policy versions
fixture hashes
session event timeline
normalized request metadata
canary event records
disposition changes
rate-limit events
classifier cost and latency
kill-switch actions
infrastructure health
operator actions
```

Avoid preserving:

- unrelated personal content;
- full request bodies when normalized features suffice;
- authentication material not generated by the honeypot;
- cross-site identifiers;
- model chain-of-thought.

Use integrity hashes and access-controlled storage.

---

## 7. Reset procedure

1. Stop ingress.
2. Export the approved evidence package.
3. Destroy all workload instances.
4. Delete ephemeral storage and queues.
5. Rotate canary registry secrets.
6. Rebuild from a pinned clean image.
7. Re-run isolation tests.
8. Re-enable only after benign probes pass.

Do not rely on cleaning a potentially modified instance in place.

---

## 8. Kill conditions

Immediately enter static terminal mode or stop the deployment when:

- prohibited outbound traffic is observed;
- an endpoint executes or attempts to execute client-supplied code;
- a production hostname, credential, or dataset is exposed;
- telemetry integrity cannot be trusted;
- CPU or memory exceeds the sustained cap;
- queue pressure affects production or control workloads;
- the classifier fails open;
- graph generation exceeds configured limits;
- canary handling is not attributable;
- the operator cannot terminate sessions deterministically.

---

## 9. Privacy and retention

Recommended defaults:

```yaml
raw_request_body_retention: disabled
normalized_event_retention_days: 30
p1_evidence_retention_days: 90
ip_storage: truncated_or_keyed
cookie_scope: honeypot_host_only
cross_site_tracking: disabled
operator_access: least_privilege
```

Adjust only with approved policy.

---

## 10. Change management

Any change to fixtures, scoring, or thresholds requires:

- a version increment;
- benign replay tests;
- adversarial replay tests;
- false-positive review;
- resource-cost comparison;
- documented rollback;
- operator notification for material policy changes.
